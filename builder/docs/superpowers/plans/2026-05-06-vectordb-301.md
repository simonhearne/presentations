````markdown
{.section}
# Quantization
## 01

---

# Why every vector DB ends up here

100M × 768 dims × 4 bytes/float = **307 GB** of raw vectors.

That's the floor. Your HNSW graph adds 30–80% on top. Replicas multiply everything.

RAM is roughly **$5/GB/month** at AWS list prices. Do the math: a billion 768-dim vectors at full precision, three replicas, is ~$50K/month *just for RAM*. Before you've served a single query.

Quantization is how you push that ceiling down. Often by 8×. Sometimes by 32×.

---

# PQ — chop and codebook

**Product Quantization.** Chop a vector into `m` sub-vectors. For each sub-vector position, learn a small codebook (typically 256 entries — one byte). Store each vector as `m` codebook indices.

A 768-dim FP32 vector is 3072 bytes raw. With `m=16, nbits=8`, it's **16 bytes** — a 192× compression ratio.

The recall cost: each sub-vector is approximated to its nearest codebook entry. Distances become approximate. The whole point of the next two slides is the knobs that control how approximate.

---

# PQ knobs

| Param | What it does | Bigger means |
|---|---|---|
| **`m`** | number of sub-vectors | finer approximation, slower build, marginally bigger code |
| **`nbits`** | bits per sub-code (codebook size = 2^nbits) | bigger codebook, better recall, more RAM for codebooks |

Compression ratio: `(d × 32) / (m × nbits)`. A 768-dim vector at `m=16, nbits=8` compresses 192×.

Reasonable starting point: `m=16, nbits=8`. Bump `m` if recall is short, drop it if RAM is tight.

There's a soft floor — past `m≈d/4` recall stops improving and build time keeps climbing. You're paying for nothing.

---

# SQ — the cheap middle ground

**Scalar Quantization.** Quantize each individual dimension to fewer bits — fp16 (2× compression, basically lossless), int8 (4× compression, ~99% recall on most embeddings).

No codebook. No build cost worth mentioning. No knobs.

Reach for SQ first. If 4× compression is enough — and for a lot of workloads it is — you're done. PQ and binary are for when SQ leaves money on the table.

The reason most production decks skip this slide: it's boring. Boring is the point.

---

# Binary + Hamming

One bit per dimension. Distance becomes Hamming distance, computed with a CPU `popcount` instruction — single-cycle on modern hardware.

768-dim FP32 → 96 bytes. **32× compression.**

The catch: recall craters on its own. Vanilla binary on dense embeddings tops out around 0.7–0.85 recall@10 — unusable for most applications.

Almost nobody runs binary as the final stage. The pattern is: binary for fast candidate generation (top-1000), then exact rescore. We'll come back to that two slides from now.

---
````

```javascript
import { writeFileSync } from "node:fs";

const W = 1200, H = 520;
const PAD = 20;
const PLOT_W = W - 2 * PAD, PLOT_H = H - 2 * PAD;
const GRID_N = 8;
const CELL_W = PLOT_W / GRID_N, CELL_H = PLOT_H / GRID_N;

function seedRandom(seed) {
  let s = seed >>> 0;
  return () => {
    s = (s * 1664525 + 1013904223) >>> 0;
    return s / 0x100000000;
  };
}

function gauss(rand) {
  return (rand() + rand() + rand() - 1.5) * 2;
}

function genPoints(rotateDeg, seed) {
  const rand = seedRandom(seed);
  const rad = (rotateDeg * Math.PI) / 180;
  const cos = Math.cos(rad), sin = Math.sin(rad);
  const points = [];
  for (let i = 0; i < 300; i++) {
    const u = gauss(rand) * 2.6;
    const v = gauss(rand) * 0.5;
    const x = u * cos - v * sin;
    const y = u * sin + v * cos;
    points.push([x, y]);
  }
  return points;
}

function projectToPx(points) {
  return points.map(([x, y]) => [
    PAD + ((x + 6) / 12) * PLOT_W,
    H - PAD - ((y + 6) / 12) * PLOT_H,
  ]);
}

function occupiedCells(pxPoints) {
  const cells = new Set();
  for (const [px, py] of pxPoints) {
    const cx = Math.floor((px - PAD) / CELL_W);
    const cy = Math.floor((py - PAD) / CELL_H);
    if (cx >= 0 && cx < GRID_N && cy >= 0 && cy < GRID_N) {
      cells.add(`${cx},${cy}`);
    }
  }
  return cells;
}

function buildSvg(rotateDeg, label) {
  const points = genPoints(rotateDeg, 42);
  const px = projectToPx(points);
  const cells = occupiedCells(px);

  const rects = [];
  for (let cx = 0; cx < GRID_N; cx++) {
    for (let cy = 0; cy < GRID_N; cy++) {
      const occupied = cells.has(`${cx},${cy}`);
      const x = PAD + cx * CELL_W;
      const y = PAD + cy * CELL_H;
      rects.push(
        `<rect x="${x.toFixed(1)}" y="${y.toFixed(1)}" width="${CELL_W.toFixed(1)}" height="${CELL_H.toFixed(1)}" ` +
          `fill="${occupied ? "#1f6feb" : "transparent"}" fill-opacity="${occupied ? 0.14 : 0}" ` +
          `stroke="#cbd5e1" stroke-dasharray="2,4" />`,
      );
    }
  }

  const dots = px
    .map(([x, y]) => `<circle cx="${x.toFixed(1)}" cy="${y.toFixed(1)}" r="3" fill="#0d2b5c" fill-opacity="0.75" />`)
    .join("\n  ");

  const counter = `${cells.size} / ${GRID_N * GRID_N} cells used`;

  return `<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 ${W} ${H}" width="${W}" height="${H}">
  <rect width="${W}" height="${H}" fill="transparent" />
  ${rects.join("\n  ")}
  ${dots}
  <text x="24" y="40" font-family="Inter, system-ui, sans-serif" font-size="22" font-weight="600" fill="#0d2b5c">${counter}</text>
  <text x="24" y="${H - 24}" font-family="Inter, system-ui, sans-serif" font-size="14" fill="#64748b">${label}</text>
</svg>
`;
}

writeFileSync(
  new URL("./before.svg", import.meta.url),
  buildSvg(0, "Lopsided variance — most grid cells wasted on empty space."),
);
writeFileSync(
  new URL("./after.svg", import.meta.url),
  buildSvg(-45, "After rotation — variance spread evenly, cells carry information."),
);

console.log("wrote before.svg and after.svg");
```

````markdown
# Why quantization wastes bytes

PQ assumes each chunk carries equal information. Real embeddings are lopsided — most variance lives in a few directions, the rest is nearly constant. Codebook entries on the constant chunks are bits well spent. Codebook entries on the high-variance chunks are bits well spent too. Bits split *evenly* across both is the problem.

<img src="./before.svg" alt="" class="figure" />

> Imagine throwing darts at a board. If 90% of darts land along one diagonal stripe, a regular grid wastes most of its squares on empty space. You're spending bits on regions of the board nothing lives in.

---

# Rotate first, quantize second
````

````markdown
# Two-stage rerank — recall back at 1/8 RAM

The pattern that ships in production:

1. **Stage 1** — quantized search over the full corpus. Returns top-N candidates (N = 100–500). Fast, cheap, approximate.
2. **Stage 2** — rescore the candidates against the **full-precision** vectors (kept on disk or in a separate cache). Sort by exact distance, return top-k.

The rescore touches `N` vectors, not the whole corpus. The full-precision storage cost is amortized — you only need fast access to the candidates that survived stage 1.

Net effect: search runs at PQ latency, recall lands within 1% of FP32. The 1/8 RAM is real, the recall hit is not.

This is what most production systems do, even when their docs don't say so.

|---|---|---|---|---|

# Quantization pitfalls

- **Re-quantizing on every model swap.** The codebook is trained against your data's geometry. Change embedding model → throw it away → retrain. Bake this into your pipeline.
- **Codebook drift on streaming ingest.** PQ codebooks are trained once. As the data distribution drifts, recall slowly degrades. Re-train periodically; treat the codebook like an index.
- **Recall measured on training data only.** A frozen ground-truth set (same as 201's recall-regression alert) is the only reliable signal that your quantization is still healthy in production.
````

````markdown
# The storage hierarchy

Per-byte costs are the whole game. Three orders of magnitude separate the top of the hierarchy from the bottom. Where each vector lives is the question Section 2 answers.

---

# Hot tier — HNSW or IVF in RAM

The 201 default. Best latency (~5–20ms p99 for HNSW). Worst $/GB.

Reach for it when the working set fits — and pay attention to *working set*, not raw vector count. A billion-vector collection with a small frequently-queried hot subset doesn't need a billion vectors in RAM.

The trap: your RAM bill grows linearly with replicas. Three replicas on 100M vectors at fp32 is 900 GB resident, every minute of every day. Replicas multiply the bill, not just the QPS ceiling.

---

# Warm tier — DiskANN on NVMe

The 201 introduction, now with the cost story attached.

PQ-compressed summary in RAM (~10–30 GB for 100M vectors). Vamana graph + full-precision vectors on NVMe (~300 GB). Search traverses the graph on disk; the in-RAM PQ codes do the heavy filtering before the I/O.

Cost: ~10× cheaper per byte than hot. Latency: ~30–80ms p99. Trade-off worth taking when your latency budget allows.

NVMe isn't a slow tier — gp3 SSD reads at ~125µs. The "slow" feeling is purely about hops vs RAM, not the disk itself.

---

# Cold tier — segments on object store

S3 / GCS-backed segments, fetched on first query. Latency: hundreds of ms to seconds for the first cold pull, then warm in the local cache.

Cost: ~250× cheaper per byte than RAM. Hard to beat for the long tail of vectors that are stored "because we might need them."

The honest pattern: most production systems don't need this. It's for archival corpora (legal hold, compliance retention, log analytics) where 99% of vectors will never be queried, but might be.

---

# Access patterns drive placement

Real query distributions are Zipfian. The top 10% of vectors absorb 80–90% of the queries. The bottom 50% absorb less than 1%.

This is the lever. Don't size the hot tier for "all vectors." Size it for the **hot fraction** — the queryable working set.

Worked example: 100M vectors, 10% hot. Hot tier holds 10M; cold tier holds 90M on object store. Total RAM: 1/10 of the naïve sizing. Latency on the hot path is unchanged. The cold path is slower, and you queue or batch around it.

The hot fraction is the single biggest cost lever after quantization.

---

# Query path across tiers

```dot
rankdir=LR
Q [label="Query"]
H [label="Hot cache\n(RAM)" fillcolor="#e8f0fe"]
W [label="Warm cache\n(NVMe)" fillcolor="#e7f7ee"]
C [label="Cold store\n(S3)" shape=cylinder]
R [label="Top-k"]
Q -> H [label="hit\n<10ms"]
H -> R
H -> W [label="miss\n+30ms" style=dashed]
W -> R
W -> C [label="miss\n+200ms" style=dashed color="#dc2626"]
C -> W [label="promote"]
W -> R
```

Each miss adds a hop. The cost of a *single* cold pull is fine. The cost of a steady-state cold-miss rate is what kills you.

---

# Time-based tiering

When age is a reliable cold-signal — logs, events, chat history — tier on it.

Operationally: declare segments older than N days warm, older than M days cold. The pipeline runs nightly. No query-pattern analysis required.

When it works: monotonically-arriving data with a recency-driven access pattern.

When it doesn't: corpora where stale data is queried as much as fresh data (knowledge bases, product catalogues). Tier on access pattern there, not age.

---

# mmap — the sleeper option

Memory-map the index file. Let the OS page in hot pages, evict cold ones. Zero application-level cache code.

When it works:
- Read-mostly workloads
- Working sets that comfortably fit in OS page cache
- Cold tail is genuinely cold

When it lies:
- Cold-page latency masquerades as "healthy averages" — p99 looks fine until p99.9 reveals the misses
- Heavy ingest invalidates pages and triggers thrash
- Cross-AZ deployment makes page-cache locality unpredictable

mmap is a good "first move before reaching for explicit tiering." If it's enough, ship it.

---

# Tiered storage pitfalls

- **Tail latency from cold pulls.** A single S3 round-trip is 100–300ms; in a fan-out query it stretches the whole p99. Pre-warm aggressively.
- **Over-eager eviction during ingest spikes.** Bulk loads can push hot data out of cache. Partition the cache by ingest vs query if your workload is bursty.
- **Tiering interacts badly with replication.** Each replica fetches independently from object store on a cold miss — a single popular cold key can stampede your S3 bill.
````

````markdown
# The cost formula

```text

    RAM:      vectors × bytes_per_vector × replicas × $RAM/GB-month

Whiteboard math. Five terms. Most spreadsheets stop after the first two.

Same dataset, three quantization configs, three monthly bills:

The single biggest lever in the bill. You read Section 1 for this slide.

# Knob 2 — tiering

The hot fraction is the lever. Holding everything else constant, varying the hot fraction:

| Hot fraction | RAM (100M @ SQ8) | Cold (NVMe) | $/month total |
|---|---|---|---|
| **100%** | 230 GB | 0  | ~$3,500 |
| **50%**  | 115 GB | 95 GB SSD | ~$1,800 |
| **20%**  | 46 GB  | 152 GB SSD | ~$760 |
| **10%**  | 23 GB  | 171 GB SSD | ~$430 |

Cutting hot fraction 10× cuts cost ~8×. Latency on the hot path is unchanged. This is where Section 2 pays for itself.

---

# Knob 3 — replicas

The 201 math, now in dollars. From slide 24 of 201: 4 replicas to hit 5000 QPS at p99 50ms across three AZs.

If your single-replica RAM bill is $3500/mo (SQ8, 100M, 1 replica), 4 replicas is **$14,000/mo**. The 4× isn't a typo — replicas multiply the whole RAM bill, not just the QPS ceiling.

The fix isn't fewer replicas (HA is real). The fix is sizing replicas to *actual peak QPS, not your worst-fear projection*. Most clusters are massively over-replicated, by a factor of 1.5–3×.

Measure your actual p99 QPS over the past quarter. Right-size from data, not anxiety.

---

# Knob 4 — managed vs self-hosted

The honest comparison Zilliz Cloud earns on technical merit, not marketing:

| Line item | Self-hosted on AWS | Zilliz Cloud |
|---|---|---|
| **Per-node QPS** | OSS Milvus baseline | **Cardinal** — typically **3× higher**, sometimes 10× |
| **Index tuning labour** | ~1 SRE-week/quarter | **AUTOINDEX** — zero |
| **List price per node** | lower | higher |
| **Operational complexity** | yours | bundled |

Cardinal compresses the `per_node_QPS` denominator in the cost formula — fewer nodes hit the same QPS target. AUTOINDEX zeroes out the labour line.

Net: managed list price is higher, but `$/query` is often **lower** once both effects are honestly accounted for. The next slide's calculator lets you run the math yourself with your numbers.

# Cost engineering pitfalls

- **Egress fees** — cross-AZ replication traffic is metered. A 100M-vector replica sync can be hundreds of dollars. Mind the topology.
- **Idle dev clusters** — non-prod environments are 30–50% of many AI bills. Tear them down on schedule, not "when someone remembers."
- **`$/vector` instead of `$/query`** — the dataset-storage cost is rarely the binding constraint; the QPS-serving cost is. Optimize the right denominator.
- **Pricing tiers and committed-use discounts** — list prices are list prices. Real bills depend on contracts. Get committed-use discounts on the steady-state nodes.

---

# If you only do three things

1. **Quantize aggressively.** SQ8 first; OPQ when SQ8 isn't enough. Pay for FP32 only at benchmark time.
2. **Tier by access pattern.** The hot fraction is the lever. Most vectors are cold; let them be cold.
3. **Right-size replicas to actual QPS.** Not peak fear. Measure for a quarter, then size.

The three together routinely cut a vector DB bill by 5–10×. Without them, you're shipping the toy from 101 at production scale and wondering why the bill is what it is.
````
